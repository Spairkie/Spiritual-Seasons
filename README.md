# Spiritual Seasons — Daily Devotional PWA

A 120-day personal devotional Progressive Web App by **Dr. Jacqueline Ghee, MSW, MPCC, DMIN**.

Discover your spiritual season (Winter · Spring · Summer · Autumn) through a short quiz, then journey through 30 daily devotionals per season — each with scripture, a reflection prompt, and your private journal.

---

## Features

- **Season quiz** — Discover whether you're in Winter, Spring, Summer, or Autumn
- **120 devotionals** — Scripture reference, full quote, and reflection prompt for every day
- **Journal** — Private text entries, auto-saved with 2-second debounce
- **Progress tracking** — Streaks, milestones, and season completion rings
- **Wellness tools** — Meditation timer, guided breathing, ambient sounds
- **Offline first** — Full PWA with service worker; works without internet
- **Installable** — Add to home screen on iOS, Android, and desktop
- **No account required** — All data stays on your device

## Tech Stack

| Layer | Technology |
|---|---|

## Getting Started

### Prerequisites

- Node.js 18+
- npm 9+

### Install

```bash
npm install
```

### Development

```bash
npm run dev
```

Opens at `http://localhost:5173`. Hot module replacement is enabled.

### Build

```bash
npm run build
```

Output goes to `dist/`. The build includes:
- All TypeScript compiled and bundled
- CSS minified
- Service worker generated by Workbox
- Assets hashed for cache busting

### Preview Production Build

```bash
npm run preview
```

### Type Check

```bash
npm run type-check
```

### Tests

```bash
npm test           # watch mode
npm run test:run   # single run
npm run test:coverage  # with coverage report
```

## Design System

All visual values use CSS custom properties from `src/styles/base/variables.css`. Never use hardcoded pixels or hex colors in component CSS.

**Seasonal theming** is applied via `[data-season="winter|spring|summer|autumn"]` on `<html>`. A single CSS selector per season handles the entire app — no per-page ID duplication.

**Breakpoints:**

| Name | Width | Layout change |
|---|---|---|
| Mobile | < 768px | Bottom nav, single column |
| Tablet | ≥ 768px | Sidebar nav, 2-column layouts |
| Desktop | ≥ 1024px | Expanded sidebar with labels, 3-column |
| Wide | ≥ 1280px | Larger base font |

## IndexedDB Schema

Database name: `spiritual-seasons-db`, version `1`.

All schema names are **identical to the original app** — existing user data migrates automatically when upgrading from the old app (same origin only).

| Store | Key | Purpose |
|---|---|---|
| `user` | `'user'` | Quiz result, current day, start date |
| `journal` | day number | Text journal entries |
| `progress` | day number | Completion timestamps |
| `favorites` | day number | Saved days with notes |
| `settings` | key string | App preferences |
| `audioNotes` | day number | Audio journal recordings |
| `weeklyReflections` | week number | Weekly reflection responses |
| `streaks` | `'streaks'` | Streak counter and milestones |

## Deployment

Two deployment targets are supported simultaneously. **No build configuration change is needed between them** — `base: './'` in `vite.config.ts` produces relative asset URLs that resolve correctly from any path.

### Netlify (primary)

- **URL:** `https://spiritual-seasons.netlify.app/`
- **Config:** `netlify.toml` — `npm run build`, `publish = "dist"`, SPA redirect `/* → /index.html 200`
- **Base path:** `/` — assets resolve as `https://spiritual-seasons.netlify.app/assets/...`
- **Deploy:** Automatic on every push to `main` via Netlify's GitHub integration

### GitHub Pages (secondary)

- **URL:** `https://spairkie.github.io/Spiritual-Seasons/`
- **Config:** `.github/workflows/deploy-pages.yml` — runs `npm run build` and uploads `dist/` to Pages
- **Base path:** `/Spiritual-Seasons/` — assets resolve as `https://spairkie.github.io/Spiritual-Seasons/assets/...`
- **Deploy:** Automatic on every push to `main` via the GitHub Actions workflow
- **Routing:** Hash-based routing (`#route`) requires no server-side rewrite rules, so the GitHub Pages static host works without any `404.html` trick

### Why `base: './'` works for both

Vite's relative base emits `./assets/index.js` (not `/assets/index.js`) in `index.html`. The browser resolves relative URLs against the document's own URL:

| Deployment | `index.html` location | `./assets/...` resolves to |
|---|---|---|
| Netlify | `https://spiritual-seasons.netlify.app/index.html` | `https://spiritual-seasons.netlify.app/assets/...` ✓ |
| GitHub Pages | `https://spairkie.github.io/Spiritual-Seasons/index.html` | `https://spairkie.github.io/Spiritual-Seasons/assets/...` ✓ |

The PWA `manifest.webmanifest` and all icon paths also use `./` for the same reason.

### Adding a new deployment target

If a future target serves the app at a fixed absolute path (e.g. a custom subdomain at `/`), no changes are needed — `base: './'` already handles it. Only if the app were served at a hardcoded subdirectory with a **different** build would you need to set `VITE_BASE=/that-path/` and update `vite.config.ts` to read `process.env.VITE_BASE`.
